--- format used ---

No project format was loaded or prescribed. Format was inferred from general
engineering practice:
- Markdown heading hierarchy (h1 title, h2 sections)
- No YAML frontmatter — no indication it was required without a template
- Sections: Problem Statement, Functional Requirements, Non-Functional
  Requirements, Constraints, Assumptions, Out of Scope
- FR/NFR sequential IDs (FR-001, NFR-001, ...)
- EARS-style phrasing for event-triggered FRs ("When X, the system shall Y")
- Measurable NFRs with explicit thresholds and measurement conditions

--- process used ---

1. Parsed the raw feature description to identify:
   - Event triggers: new message, task completion, system alert → FR-001–003
   - Channel support: email, in-app → FR-004–005
   - User control: per-channel preferences → FR-006
   - Failure handling: retry/backoff — prose input was vague, so bounds
     were added (5 attempts, 10 minutes) using engineering judgment → FR-007
   - Latency: 30 seconds → NFR-001 with p99 qualifier added
   - Delivery rate: 99.9% → NFR-002, denominator made explicit
   - Throughput: 10k/min → NFR-003, referenced NFR-001/NFR-002 to avoid
     restating thresholds
   - Isolation implied by delivery rate SLA → NFR-004
2. Surfaced constraints: SMS excluded in v1 with forward-compatibility note,
   email provider constraint carried through from prose.
3. Added Assumptions for dependencies required for FRs to hold but not
   stated in the input.
4. Added "notification read/unread state" to Out of Scope as an implicit
   gap for in-app delivery.
