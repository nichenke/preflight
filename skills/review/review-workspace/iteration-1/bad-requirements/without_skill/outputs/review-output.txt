Review of: docs/requirements.md
Review type: Requirements document quality review (baseline, no skill)

## Summary

This requirements document has significant quality issues across completeness, clarity, measurability, and structural consistency. Only 2 of 11 requirements meet a reasonable quality bar. The document would not be suitable for driving implementation without substantial rework.

## Findings

### Critical

1. Missing requirement IDs (lines 12-13)
   Two functional requirements lack IDs entirely:
   - "The system should send a confirmation email after signup." (no ID, would be FR-003)
   - "Users can reset their password via the settings page." (no ID, would be FR-004)
   Missing IDs make requirements untrackable and unreferenceable in test plans, ADRs, or implementation tickets.

2. Unmeasurable non-functional requirements (lines 17-19)
   - NFR-001: "The system shall be fast." — No metric. Fast compared to what? No latency target, no throughput target.
   - NFR-002: "The system shall be secure." — No specification of security properties, compliance standards, threat model, or controls.
   - NFR-003: "The system shall have good uptime." — No SLA target (99.9%? 99.99%?), no measurement window.
   These are effectively placeholders, not requirements. They cannot be tested or verified.

3. Inconsistent requirement language (lines 12-13)
   - "should send" (FR-003 equivalent) uses advisory language instead of mandatory "shall"
   - "Users can reset" (FR-004 equivalent) describes a capability, not a requirement
   Requirements documents need consistent use of "shall" for mandatory behavior. Mixed language creates ambiguity about what is required vs. optional vs. aspirational.

### High

4. Gap in ID sequence (line 13)
   IDs jump from FR-002 to FR-005, with two unnumbered requirements in between. This suggests either IDs were lost during editing or the numbering was applied inconsistently. The gap makes it unclear whether FR-003 and FR-004 were intentionally skipped or are the unnumbered items.

5. Constraints lack IDs and measurability (lines 24-25)
   - "The system must run on AWS infrastructure." — No constraint ID. Which AWS services? Any region constraints?
   - "Budget is limited so we should keep costs low." — Not a constraint, it is a vague aspiration. No budget figure, no cost target, no unit economics threshold.

6. No traceability structure
   The document has no cross-references to architecture decisions, test plans, or design documents. Requirements exist in isolation with no way to verify coverage.

### Medium

7. Incomplete functional coverage
   The document describes a signup/user-management system but has obvious gaps:
   - No requirement for user authentication/login
   - No requirement for data storage or persistence
   - No requirement for what happens after email confirmation
   - No error handling requirements beyond FR-005
   - No requirement for password reset flow details (email link? security questions?)

8. FR-005 mixes trigger and performance (line 13)
   "When a user submits invalid data, the system shall display an error message within 2 seconds." combines a functional requirement (display error on invalid data) with a performance requirement (within 2 seconds). These should be separate requirements so they can be independently tested and traced.

9. No versioning or status metadata
   The document has a YAML frontmatter with only `type: requirements`. No version, no author, no status (draft/approved/deprecated), no last-updated date.

### Low

10. Section 3 (Constraints) is thin
    Two items, one of which is not actionable. Constraints typically cover technology stack details, regulatory requirements, team skill constraints, timeline, and integration requirements. This section needs expansion or an explicit note that constraints are tracked elsewhere.

## What works

- FR-001, FR-002, and FR-005 use proper "shall" language with clear subjects
- FR-005 includes a measurable threshold (2 seconds)
- NFR-004 is well-formed with concrete metrics (1000 concurrent users, p95 < 200ms)
- Document structure (FR / NFR / Constraints sections) is reasonable

## Recommendation

This document is in draft quality at best. Before using it to drive implementation:
1. Assign IDs to every requirement, including constraints
2. Replace vague NFRs (001-003) with measurable targets
3. Normalize all requirement language to use "shall" consistently
4. Fill functional coverage gaps (auth, persistence, full password reset flow)
5. Add document metadata (version, status, author, date)
6. Add traceability references to related architecture and test documents
