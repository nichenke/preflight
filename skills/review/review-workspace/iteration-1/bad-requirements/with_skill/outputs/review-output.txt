## Review: requirements.md
Document type: requirements | Rules: universal-rules.md, requirements-rules.md, cross-doc-rules.md

### Findings

**[Critical] missing-frontmatter-fields** (confidence: 98, UNIV-01)
Frontmatter contains only `type: requirements`. Missing required fields: status, date, owner, version. The YAML block is: `---\ntype: requirements\n---`
**Consequence:** No traceability — consumers cannot determine document currency, ownership, or lifecycle stage. Automated tooling that keys on status or version will fail silently.
**Fix:** Add status (e.g., Draft), date (ISO 8601), owner, and version fields to the YAML frontmatter.

---

**[Critical] requirements-missing-ids** (confidence: 98, REQ-R02)
Two functional requirements lack unique IDs. Line 11: "The system should send a confirmation email after signup" and line 12: "Users can reset their password via the settings page" have no FR-NNN identifier. The numbering also skips from FR-002 to FR-005, leaving a gap.
**Consequence:** These requirements cannot be traced to architecture components, test cases, or ADRs. Cross-references from other documents will fail. Agents generating implementation will miss these requirements or produce duplicate work.
**Fix:** Assign sequential FR IDs (FR-003, FR-004) to the two unidentified requirements and verify no gaps remain in the sequence.

---

**[Important] vague-nfr-no-quantification** (confidence: 98, UNIV-04)
Three NFRs use vague adjectives without quantification: NFR-001 "The system shall be fast", NFR-002 "The system shall be secure", NFR-003 "The system shall have good uptime". The words "fast", "secure", and "good" are flagged by UNIV-04.
**Consequence:** No acceptance test can be written for these NFRs. An implementation taking 30 seconds per request could claim compliance with "fast." QA and agents have no threshold to verify against.
**Fix:** Replace each vague adjective with a measurable criterion (e.g., "p95 response time under 500ms", "TLS 1.3 for all data in transit with annual penetration test", "99.9% monthly uptime").
[Convergence: also flagged by bogey-reviewer Layer 3 as unverifiable-nfrs]

---

**[Important] nfr-missing-quantitative-criteria** (confidence: 98, REQ-R04)
NFR-001, NFR-002, and NFR-003 lack quantitative acceptance criteria. NFR-004 demonstrates the expected pattern with "at least 1000 concurrent users with p95 response time under 200ms" — the other three have no equivalent specification.
**Consequence:** These NFRs are unverifiable and untestable. They provide no signal for pass/fail during implementation or delivery review.
**Fix:** Add specific, measurable acceptance criteria to NFR-001, NFR-002, and NFR-003, following the pattern established by NFR-004.

---

**[Important] non-ears-notation** (confidence: 95, REQ-R01)
Two requirements do not use EARS notation. Line 11: "The system should send a confirmation email after signup" uses "should" instead of "shall" and lacks an EARS keyword. Line 12: "Users can reset their password via the settings page" uses "can" instead of "shall" and lacks an EARS keyword.
**Consequence:** "Should" and "can" are ambiguous — they may be interpreted as optional rather than mandatory. Agents generating code may skip these requirements. This also silently violates constitution principle CONST-QUAL-01 requiring EARS patterns.
**Fix:** Rewrite using EARS notation: e.g., "When a user completes signup, the system shall send a confirmation email" and "The system shall allow users to reset their password via the settings page."
[Convergence: also flagged by bogey-reviewer Layer 3 as silent-constitution-violation]

---

**[Important] no-failure-modes** (confidence: 95, REQ-R05)
No failure modes are documented for any user journey. The document describes happy paths (signup, email validation, password reset) but does not address what happens when email delivery fails, invalid data is submitted beyond the FR-005 case, or password reset tokens expire.
**Consequence:** Implementers will not handle edge cases. Error handling will be ad-hoc or missing, leading to poor user experience and potential data integrity issues.
**Fix:** Add a failure mode subsection documenting at least one failure scenario per user journey (signup failure, email delivery failure, password reset failure).

---

**[Important] no-success-measures** (confidence: 92, REQ-R06)
No success measures section exists. REQ-R06 requires baseline, target, and measurement method for success criteria.
**Consequence:** There is no way to evaluate whether the system meets its goals after delivery. Stakeholder sign-off has no objective basis.
**Fix:** Add a Success Measures section with baseline values, targets, and measurement methods for key functional and non-functional outcomes.

---

**[Important] vague-budget-constraint** (confidence: 92, quality)
Line 26: "Budget is limited so we should keep costs low" uses "should" (not "shall") and provides no quantifiable constraint. This is in the Constraints section but reads as informal guidance rather than an actionable constraint.
**Consequence:** No cost threshold exists for architectural decisions. An agent or architect cannot evaluate whether a design meets budget requirements.
**Fix:** Rewrite as a measurable constraint: e.g., "The system shall operate within a monthly infrastructure budget of $X" or remove if not a binding constraint.

---

**[Suggestion] no-out-of-scope-section** (confidence: 90, quality)
The document has no "Out of Scope" section. The anti-patterns guide notes: "Every requirements doc should explicitly state what's excluded."
**Consequence:** Scope creep risk — without explicit exclusions, stakeholders and implementers may assume adjacent features (e.g., admin panel, analytics, multi-tenancy) are in scope.
**Fix:** Add an "Out of Scope" section listing features and capabilities explicitly excluded from this project phase.

---

**[Suggestion] orphaned-requirements** (confidence: 82, XDOC-09)
No architecture document exists to reference FR-001 through FR-005 or NFR-001 through NFR-004. No test strategy maps to these requirement IDs.
**Consequence:** Requirements exist without traceability to implementation or verification artifacts. As the project matures, requirements may be implemented inconsistently or missed entirely.
**Fix:** Create an architecture document and/or test strategy that references each FR and NFR ID.

- NFR-004 demonstrates the correct pattern for quantitative NFRs with concrete thresholds (1000 concurrent users, p95 < 200ms)
