# Preflight Plugin Design

## Overview

Convert the PM Documentation Framework from a CLAUDE.md-based git repo into a Claude
Code plugin called **preflight**. The plugin provides three skills (scaffold, new, review),
auto-loaded rules via scaffold-generated `.claude/rules/preflight.md`, and a content
directory that serves as the single source of truth for templates, rules, and reference
material.

### Design decisions resolved

- **Framework internals** live in `.preflight/` (committed to git, not ignored)
- **Project documents** live in `docs/` by default (or user-specified path)
- **Rules delivery** via scaffold-generated `.claude/rules/preflight.md` with concrete paths
- **Config** stored in `.preflight/config.yml` — skills read this to resolve the docs directory
- **Filename handling** — singletons use fixed names, sequential types (ADR, RFC) scan for
  next ID and ask for a slug, named types (interface contracts) ask for the boundary name
- **Post-creation review** — `/preflight new` runs a review subagent after writing,
  auto-fixes Errors, reports Warnings

## Auto-loaded rules file

Generated by scaffold into `.claude/rules/preflight.md`. Contains concrete paths based
on the user's docs directory choice. Must stay under 80 lines (NFR-003).

ADRs are deliberately excluded from the read-before-coding sequence. ADR decisions
should be propagated to downstream docs (requirements, architecture, constitution)
during creation — see "ADR impact propagation" below. ADRs are reference material
for "why did we decide this?", not session context.

Content:

```markdown
# Preflight — project spec rules

## Before writing code

Read these files in order — skip any that don't exist:

1. `{docs_dir}/constitution.md` — overrides everything
2. `{docs_dir}/requirements.md` — EARS requirements with FR/NFR IDs
3. `{docs_dir}/architecture.md` — system structure, patterns, components
4. `{docs_dir}/interfaces/` — contracts at boundaries you're touching

Check `{docs_dir}/decisions/adrs/` only when modifying requirements or architecture
— ADR decisions should already be reflected in those docs.

## Requirements change governance

No behavioral requirement change without an ADR. If a change to requirements.md
would cause an agent to generate different code, it needs an ADR first.
Clarifications, typo fixes, and added failure modes do not.

## EARS quick reference

| Pattern      | Keyword     | Template                                                    |
| ------------ | ----------- | ----------------------------------------------------------- |
| Ubiquitous   | (none)      | The <system> shall <response>.                              |
| Event-driven | **When**    | When <trigger>, the <system> shall <response>.              |
| State-driven | **While**   | While <precondition>, the <system> shall <response>.        |
| Optional     | **Where**   | Where <feature>, the <system> shall <response>.             |
| Unwanted     | **If/then** | If <condition>, then the <system> shall <response>.         |
| Complex      | Combined    | While <pre>, when <trigger>, the <system> shall <response>. |

## Document IDs

Assign unique IDs: FR-NNN, NFR-NNN, ADR-NNN, RFC-NNN, CONST-{CAT}-NN.
IDs are sequential and never reused.
```

`{docs_dir}` is replaced with the actual path (e.g., `docs`) when scaffold writes the file.

## Config file

`.preflight/config.yml` — minimal, read by all skills:

```yaml
docs_dir: docs
```

Default is `docs/`. Scaffold asks on first run, persists the choice. If config is
missing, skills default to `docs/`.

## Scaffold skill

Covers FR-001 through FR-009.

### Fresh project flow

1. Ask where project docs should live (default: `docs/`)
2. Create `.preflight/` with framework content:
   - Copy `content/templates/` → `.preflight/_templates/`
   - Copy `content/rules-source/` → `.preflight/_rules/`
   - Copy `content/reference/` → `.preflight/_reference/`
3. Create project docs structure in chosen directory:
   - `{docs_dir}/constitution.md` (skeleton from `content/scaffolds/`)
   - `{docs_dir}/glossary.md` (skeleton from `content/scaffolds/`)
   - `{docs_dir}/interfaces/` (empty directory with .gitkeep)
   - `{docs_dir}/decisions/adrs/adr-001-use-preflight.md` (from `content/scaffolds/`)
   - `{docs_dir}/decisions/rfcs/` (empty directory with .gitkeep)
4. Write `.preflight/config.yml` with docs_dir
5. Write `.claude/rules/preflight.md` with concrete paths
6. Report everything created

### Update existing flow (FR-008)

1. Detect `.preflight/` exists
2. Compare framework files (`_templates/`, `_rules/`, `_reference/`) against plugin source
3. Report diffs to user, ask which to accept
4. Never touch project docs (FR-009) — anything in `{docs_dir}/`
5. Update `.claude/rules/preflight.md` if the plugin version changed

### Protected files (FR-009)

Scaffold never overwrites:

- `{docs_dir}/constitution.md`
- `{docs_dir}/glossary.md`
- `{docs_dir}/requirements.md`
- `{docs_dir}/architecture.md`
- `{docs_dir}/test-strategy.md`
- Anything in `{docs_dir}/interfaces/`
- Anything in `{docs_dir}/decisions/`

## New skill

Covers FR-010 through FR-016.

### Entry flow

1. Read `.preflight/config.yml` for docs_dir
2. If doc type not specified, present choices: requirements, adr, rfc, architecture,
   interface-contract, test-strategy, constitution
3. Route to the appropriate elicitation flow

### Filename handling

| Category   | Doc types                                                         | Behavior                                                    |
| ---------- | ----------------------------------------------------------------- | ----------------------------------------------------------- |
| Singleton  | requirements, architecture, constitution, glossary, test-strategy | Fixed filename, no prompt                                   |
| Sequential | ADR, RFC                                                          | Scan for next ID, ask for slug → `adr-003-auth-approach.md` |
| Named      | Interface contract                                                | Ask for boundary name → `payment-gateway.md`                |

### Elicitation flows

Each flow reads its template from `${CLAUDE_PLUGIN_ROOT}/content/templates/` to
understand structure, then walks the user through guided sections:

| Doc type           | Guided sections                                                                                                                        |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| Requirements       | problem → personas → journeys (with failure modes) → EARS FRs → NFRs → constraints → assumptions → success measures → out-of-scope     |
| ADR                | context → decision drivers → ≥2 options with pros/cons → decision → consequences → confirmation criteria → impact propagation           |
| RFC                | executive summary → problem (with evidence) → scope → proposed solution → ≥1 alternative → rollout/rollback → risks → success criteria |
| Architecture       | requirements overview → context diagram → strategy → building blocks                                                                   |
| Interface contract | protocol → endpoints/operations → SLA → error handling                                                                                 |
| Test strategy      | pyramid levels → acceptance mapping → environments                                                                                     |
| Constitution       | preamble → categories → principles (imperative, testable)                                                                              |

### ADR impact propagation

After an ADR is written, the new skill runs an impact propagation step:

1. Read existing downstream docs (requirements, architecture, constitution, interfaces)
2. Identify which docs need updates to reflect the ADR's decision and consequences
3. For each affected doc, propose specific changes (new FRs/NFRs, architecture updates,
   constraint additions)
4. Present the propagation plan to the user for approval
5. Apply approved changes to downstream docs
6. Flag any ADR consequences that can't be traced to a downstream doc — these are
   decisions that would be invisible without reading the ADR, which is the failure
   mode we're preventing

This makes ADRs write-mostly: the decision is captured once, its effects are carried
to the docs that agents actually read, and the ADR itself becomes archival reference.

### Post-creation

1. Populate YAML frontmatter (status: Draft, date, owner, version: 0.1.0)
2. Assign next sequential ID where applicable (FR-016)
3. Write the file to the correct location
4. Dispatch review subagent — reads applicable rules from
   `${CLAUDE_PLUGIN_ROOT}/content/rules-source/`, universal rules, and cross-doc rules
   (uses plugin source, not `.preflight/_rules/`, because the project copy may not
   reflect the latest plugin rules or may not exist yet in a fresh scaffold)
5. Auto-fix any Error-severity findings
6. Report what was created and any Warning-severity findings for the user to consider

## Review skill

Covers FR-017 through FR-020.

### Entry flow

1. Read `.preflight/config.yml` for docs_dir
2. If file not specified, ask which document to review (or accept a path argument)
3. Identify doc type from frontmatter or infer from filename/location

### Review process

1. Read the target document
2. Load rules from `.preflight/_rules/` (the project's committed copy):
   - Type-specific rules (e.g., `requirements-rules.md`)
   - `universal-rules.md`
   - `cross-doc-rules.md` — when the document references other documents
3. Check each rule against the document
4. Group findings by severity: Error first, then Warning
5. Each finding includes: rule ID, what's wrong, specific fix suggestion
6. Zero Errors → report as passing (FR-020)

Review reads from `.preflight/_rules/` (the committed project copy), not from the
plugin source — this honors any project-specific rule customizations.

Review is advisory — it does not modify the document or block anything.
