# Review: ADR-001 — Skip CC plugin packaging for PAI

---

## Checklist-reviewer pass

### Findings

**[Important] decision-passive-voice** (confidence: 92, ADR-R02)
The Decision Outcome section uses third-person passive framing: "Proposed option: 'Direct overlay', because it provides a single deployment model covering 100% of PAI's installation surface without dependency on an immature plugin API." ADR-R02 requires the Decision section to use active voice ("We will..."). The current framing reads as a council recommendation, not a committed decision.
**Consequence:** Agents and humans reading this ADR as authoritative may treat the decision as less definitive than intended, or may re-argue it at implementation time.
**Fix:** Replace the Decision Outcome opening with: "We will use Direct overlay — install PAI to `~/.claude/` via bootstrap script, with no CC plugin involvement."

---

**[Suggestion] context-advocacy-framing** (confidence: 82, ADR-R02)
The Context section is meant to be value-neutral (describing forces, not advocating), but the phrase "Evidence from upstream issues reinforces the gap" introduces the author's conclusion as framing within the neutral description. Upstream issues #901 and others are cited, which is good, but the word "reinforces" signals that the author is building a case, not describing a constraint.
**Consequence:** Future reviewers and agents may accept the framing without independently evaluating whether the cited issues are conclusive, or may not notice that the Context is already pointing at the conclusion.
**Fix:** Reframe to factual description: "Upstream issues document known gaps: settings.json MCP config is silently ignored when placed in plugin scope (upstream #901); the plugin API provides no mechanism for managing CC's own configuration files."

---

### Strengths

1. **Validation gate structure is rigorous.** The Confirmation section does not just list criteria — it blocks advancement to Accepted until a spike proves the model end-to-end. The revisit criteria (three conditions that must all be true) provide a concrete, future-facing re-evaluation trigger. This is an above-average ADR discipline on an immature-platform decision.

2. **Four options with symmetric analysis.** All four options receive genuine pros and cons, including the chosen option's negatives. The "Neutral" consequence item for marketplace distribution is honest scoping rather than pretending the tradeoff doesn't exist.

3. **Upstream issue citations are concrete.** Issue numbers (#901, #893, #830, #650) are cited rather than vague "known limitations" — this makes the context verifiable against the actual upstream issue tracker.

**Summary: 0 Critical, 1 Important, 1 Suggestion**

## Bogey-reviewer pass

### Section 1: Layer 1 — Cross-doc assumption conflicts

**Conflict 1: "Single operation" tightens FR-023's "single source"**

- ADR-001 Decision Driver: "PAI's deployment model must install everything PAI needs — system code, CC configuration, shell integration, and daemon setup — as a single operation."
- FR-023 (requirements.md): "The system shall support installation to a new environment from a single source without manual reconstruction."

FR-023 constrains the *source* (one place to pull from), not the *mechanism* (one step to execute). The ADR's decision driver adds "single operation" as a hard eliminatory constraint — this is what makes the Hybrid option fail ("creates two deployment models"). A sequenced two-step install drawing from one source repo would satisfy FR-023 while failing the ADR's driver. The ADR has silently tightened the requirement.

Classification in ADR: HARD (presented as "must").
Classification derivable from FR-023: SOFT — FR-023 permits multi-step installation from a single source.

This is a SOFT constraint promoted to HARD without acknowledgment. The design preference is coherent (single operation is better than two steps for diagnostic simplicity), but framing it as "must" derived from requirements is not accurate.

---

**Conflict 2: 60% plugin coverage figure treated as established fact**

- ADR-001 Context: "Analysis of the CC plugin API surface shows it can express approximately 60% of PAI's installation footprint."
- Constitution CONST-DEC-01: "All technical decisions require code-level verification. Research findings are hypotheses until tested with spikes."

The 60% figure is used as a load-bearing decision input but is sourced from "analysis" with no spike citation. At "Proposed" status with a spike gate, this is within the rules — the ADR acknowledges the figure requires validation before Accepted. However, the 60% figure is also used in the "Full CC plugin" and "Hybrid" option analysis without caveat. If the actual figure is 75% or 80%, the hybrid option's viability changes materially.

Classification: ASSUMPTION — stated as fact, not verified by spike.
This is not a hard conflict given the ADR's Proposed status, but the load-bearing assumption should be flagged as unvalidated within the option analysis.

### Section 2: Layer 2 — Hypothesis investigation

**H1 — Unverifiable requirement**

**Finding — fires.**

Two validation criteria in the Confirmation section cannot be verified as satisfied or violated by an executing agent:

1. "All PAI capabilities (hooks, skills, commands, agents, CLAUDE.md generation) function identically whether installed by bootstrap or by manual file placement."

   "Function identically" provides no enumeration of what to test. A spike agent would have no basis for determining pass vs. fail — it could test hooks and miss CLAUDE.md generation, declare the criterion met, and be wrong. No test cases, no test list, no acceptance definition.

2. "Upgrade cycle (upstream merge, test in source repo, bootstrap to sandvault) completes without runtime disruption."

   "Without runtime disruption" is undefined. A one-time error on first run after bootstrap: disruption or not? A 5-second hook failure: disruption or not? A running session that survives but misses a new capability: disruption or not? The criterion cannot be deterministically evaluated.

These criteria gate the ADR's transition to Accepted. If they are unverifiable, the ADR either never advances (blocked on ambiguous criteria) or advances on an underdefined interpretation of "pass."

---

**H2 — Silent conflict**

**Partial finding — reduced to documentation gap after steelman.**

Constitution CONST-BUILD-02 requires an RFC for build decisions. ADR-001 recommends building custom bootstrap infrastructure. RFC-002 is referenced in "More information" as "Plugin vs overlay deployment model" — this is the natural vehicle for CONST-BUILD-02 compliance.

However, the ADR does not explicitly acknowledge CONST-BUILD-02 or state that RFC-002 satisfies it. An agent reading only the ADR cannot confirm CONST-BUILD-02 has been met without independently locating and reading RFC-002. The gap is a documentation linkage issue, not a genuine conflict.

**Steelman holds for full conflict — the RFC-002 reference is sufficient.** Not escalating to a validated finding. Flagging as a low-severity linkage gap for awareness.

---

**H3 — False hard constraint**

**Finding — fires.**

The "single operation / single deployment model" framing is the primary mechanism by which the ADR eliminates the Hybrid option:

> "Bad, because it creates split-brain architecture — two deployment models, two update paths, two diagnostic paths for a single system."

This evaluation rests entirely on the "single operation" constraint. But FR-023 does not require single operation — it requires single source. A hybrid approach using one source repo with a sequenced install (plugin step + bootstrap step) would satisfy FR-023. The ADR forecloses this without acknowledging that "single operation" is an ADR-level design choice, not a requirement-derived constraint.

The design preference for single operation is coherent and defensible. The problem is that it is not labeled as a design choice — it is framed as a hard constraint ("must"), which makes the elimination of Hybrid look like a requirements-driven conclusion when it is actually a policy decision embedded in the decision drivers.

### Section 3: Layer 3 — Validated findings

**[Important] unverifiable-validation-criteria** (confidence: 88, structural)
The Confirmation gate that blocks transition from Proposed to Accepted includes two criteria that cannot be deterministically verified:

- "All PAI capabilities (hooks, skills, commands, agents, CLAUDE.md generation) function identically whether installed by bootstrap or by manual file placement." — no test enumeration, no pass/fail definition.
- "Upgrade cycle...completes without runtime disruption." — "runtime disruption" is undefined; no threshold, no observable state that constitutes violation.

**Consequence:** The spike agent executing validation has no deterministic basis for closing the gate. The ADR either stalls at Proposed indefinitely (awaiting criteria that can't be met or assessed), or advances on a vague interpretation that may not catch real installation failures. This is the highest-impact finding because it directly blocks the decision lifecycle.
**Fix:** Replace both criteria with enumerated, observable tests. Example replacements: (1) "Bootstrap installs all of: hooks, skills, commands, agents, CLAUDE.md template, rules files, statusline config, shell aliases, and LaunchAgent — verified by a checklist test that probes each installed artifact's presence and basic invocability." (2) "Upgrade cycle completes with zero errors in the bootstrap script and no session crashes on the first post-upgrade CC launch."
[Convergence: also flagged by Layer 2 H1]

---

**[Important] single-operation-constraint-undeclared** (confidence: 87, structural)
The ADR's Decision Driver states PAI's deployment model "must install everything...as a single operation" and the option analysis eliminates Hybrid on this basis. However, FR-023 requires installation "from a single source without manual reconstruction" — not from a single operation. The "single operation" constraint is an ADR-level design choice, not a requirements-derived hard constraint. Presenting it as "must" without acknowledging the derivation makes the Hybrid option look like a requirements violation when it is actually eliminated by a policy preference.

**Consequence:** If the plugin API matures or if a future agent re-evaluates this decision, the "single operation" constraint will be treated as a hard requirement when it is actually revisable. This blocks valid reconsideration and creates a false hard dependency in the revisit criteria.
**Fix:** In the Decision Drivers section, reframe "Full surface coverage" to make the design choice explicit: "We require a single deployment operation (design preference beyond FR-023's single-source requirement) because split deployment creates split diagnostic paths."
[Convergence: also flagged by Layer 1]

---

**[Suggestion] unvalidated-60-percent-figure** (confidence: 85, structural)
The 60% plugin coverage figure ("Analysis of the CC plugin API surface shows it can express approximately 60% of PAI's installation footprint") is used as a load-bearing input in the Context, in the option analysis, and in the Hybrid option evaluation — but it is sourced from "analysis" with no spike citation. The figure is also referenced in the revisit criterion ("CC plugin API covers 90% or more of PAI's installation surface (currently ~60%)"). If the actual coverage is materially higher, the hybrid option's viability and the revisit threshold change.

**Consequence:** If the 60% figure is wrong, the ADR's option analysis and revisit criteria are calibrated against a false baseline. An agent implementing the revisit criteria would apply the wrong threshold.
**Fix:** Add a note in Context citing the source of the 60% figure (e.g., the design doc or RFC-002 analysis) and add a caveat that the figure is an estimate requiring spike confirmation. In the revisit criteria, add "current ~60% figure validated by [source/spike]" or note it as estimated.
[Convergence: also flagged by Layer 1]

---

**[Suggestion] context-advocacy-framing** (confidence: 82, structural)
The phrase "Evidence from upstream issues reinforces the gap" in the Context section introduces interpretive framing ("reinforces") in a section meant to be value-neutral. This is the same finding produced by the checklist-reviewer (ADR-R02).
**Consequence:** Mild — future readers may not independently evaluate whether the cited issues are conclusive evidence or merely corroborating data points.
**Fix:** Replace "reinforces the gap" with factual description: "Upstream issues document known limitations in the current plugin API." (See checklist-reviewer finding above for full fix.)
[Convergence: also flagged by checklist-reviewer pass]
