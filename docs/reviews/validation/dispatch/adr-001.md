## Review: adr-001-terminal-confirmation.md
Document type: adr | Rules: universal-rules.md, adr-rules.md, cross-doc-rules.md

### Findings

**[Important] retroactive-justification** (confidence: 87, quality)
The frontmatter states `"Seed content from proj-handoff ADR-001"` and the Decision section notes `"systemMessage was discovered during implementation to be a cleaner equivalent using the proper API."` This indicates the ADR was written after the implementation decision was already made, fitting the anti-pattern: "Writing an ADR after the code is shipped to paper over a gut call."
**Consequence:** Rejected alternatives may not have been evaluated fairly at the time the decision was actually made — the ADR becomes a rationalization rather than a decision record, reducing trust in the options analysis.
**Fix:** Add a note in Context acknowledging the decision was made during implementation and this ADR captures it retroactively, or restructure the narrative to reflect the actual decision sequence.

---

**[Suggestion] rfc-reference-without-id** (confidence: 88, UNIV-02)
The Decision section states `"The original RFC considered stderr (Option 1)"` — this references an RFC by description rather than by document ID (e.g., RFC-NNN).
**Consequence:** A reader or agent cannot trace this reference to the actual RFC document, breaking cross-doc traceability.
**Fix:** Replace the prose reference with the RFC's document ID (e.g., "RFC-001 considered stderr").

---

**[Suggestion] vague-adjective-cleaner** (confidence: 85, UNIV-04)
The Decision section states `"systemMessage was discovered during implementation to be a cleaner equivalent"` — "cleaner" is a vague adjective without quantification or specific criteria.
**Consequence:** A future reader cannot evaluate what "cleaner" means in this context — fewer lines of code? Fewer failure modes? Better API ergonomics? The rationale loses precision.
**Fix:** Replace "cleaner equivalent" with specific criteria (e.g., "a single-field equivalent that avoids stderr rendering inconsistencies").

---

**[Suggestion] context-mild-advocacy** (confidence: 82, ADR-R02)
The Context section uses the framing `"The gap: is the user notified before Claude's first response?"` — this rhetorical question implies a deficiency rather than neutrally describing forces. ADR-R02 requires Context to be value-neutral.
**Consequence:** The Context section subtly steers the reader toward a predetermined conclusion rather than presenting forces for independent evaluation.
**Fix:** Reframe as a neutral statement: "Open question: whether and how the user is notified before Claude's first response."

---

**[Suggestion] title-not-imperative** (confidence: 85, quality)
The title is `"Terminal confirmation mechanism — systemMessage"` — a noun phrase. The ADR anti-patterns guidance states titles should be "short imperative verb phrases" (e.g., "Use systemMessage for terminal confirmation").
**Consequence:** Noun-phrase titles don't communicate the decision at a glance in an ADR index — readers must open the document to understand what was decided.
**Fix:** Rename to an imperative form: "Use systemMessage for terminal confirmation."

---

**[Suggestion] load-bearing-rendering-order-assumption** (confidence: 83, structural)
The ADR states `"CC renders this in the terminal as a native hook acknowledgment before Claude's first response"` and lists `"Must appear before Claude's first token (pre-keypress)"` as a decision driver. The rendering-order behavior of `systemMessage` is treated as a fact but is not verified or cited — it is a load-bearing assumption about CC internals.
**Consequence:** If CC changes the rendering order of `systemMessage` relative to the first response token, the primary decision driver is violated and the chosen option may no longer satisfy the requirement.
**Fix:** Add a note in Consequences or Context citing where this rendering-order behavior is documented, or flag it as an assumption that should be validated.

---

**[Suggestion] no-degradation-behavior-stated** (confidence: 82, structural)
The ADR introduces `systemMessage` as a new hook output path but does not address what happens if `systemMessage` emission fails or is ignored by the terminal. The constitution requires (CONST-DEGRADE-03): `"Hook errors are caught and exit 0 with stderr-only warnings. A crash in dispatch must never prevent Claude Code from starting or responding."`
**Consequence:** An implementer has no guidance on how to handle `systemMessage` failures, risking either a silent failure with no user notification or an unhandled error that violates the constitution's degradation requirements.
**Fix:** Add a Consequence item noting that `systemMessage` failure should degrade gracefully per CONST-DEGRADE-03, with the hook continuing without terminal confirmation.
