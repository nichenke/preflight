# Review: RFC-001 — Specialized Reviewer Agents for Document Quality Assessment

## Summary

**1 Critical, 1 Important, 1 Suggestion**

---

## Findings

**\[Critical] confidence-tier-conflict** (confidence: 97, structural)

The RFC's confidence-to-severity mapping in the Proposed Solution section is incompatible with the confidence-to-severity mapping already implemented in `agents/reviewers/checklist-reviewer.md`.

RFC defines (lines 91–104):

> "80–89: Report as Suggestion / 90–99: Report as Important / 100: Report as Critical"

`checklist-reviewer.md` defines:

> "Error rule + confidence ≥95 → Critical / Error rule + confidence 80–94 → Important / Warning rule + confidence ≥90 → Important"

The RFC reserves Critical for confidence 100 only. The checklist-reviewer fires Critical at confidence ≥95 on Error-severity rules. These tiers produce different outputs for the same finding — a confidence-95 Error finding is Critical under the agent's scheme but Important under the RFC's scheme.

**Consequence:** Any agent implemented against the RFC spec will disagree with checklist-reviewer.md on Critical vs. Important classification. The review output contract — which downstream users, scripts, and ADR-004 will depend on — is undefined. Both documents claim to specify the same system.

**Fix:** Align the RFC's confidence tier table with the checklist-reviewer.md implementation, or update the agent to match the RFC. :comment[The RFC is the authoritative specification; the likely correct fix is updating the RFC's tier table to match the implemented agent, then adding a note that the scheme maps severity + confidence jointly (not confidence alone).]{#comment-1774990627183 text="approved -- unless there**SQUOTE**s a timing/adr gap here. where the implemented agent was done later, but this gap is showing us a missing decision? Where do we define the requirements/rules for quality - do we have some larger gap here ?"}

**\[Important] missing-nfr-for-reviewer-accuracy** (confidence: 90, RFC-R06)

The Problem Statement claims reviewer agents will catch violations that the current monolithic skill misses. No NFR in the RFC or requirements.md establishes a minimum quality bar for the agents themselves as reviewer accuracy targets for phase 1 output.

The RFC states:

> "single-prompt evaluation misses violations that a focused agent would catch"

The Success Criteria for phase 1 are entirely structural:

> "All 7 reviewer agent files exist with valid frontmatter and pass plugin structure validation (NFR-006)"

Phase 2 criteria do include "≥85% rule-following accuracy... false positive rate ≤15%" — but these are gated on a follow-on ADR and do not bind phase 1 agent quality.

**Consequence:** Phase 1 agents ship with no quality floor. A reviewer agent that produces worse output than the current skill satisfies all phase 1 success criteria. The RFC's core claim — that specialization improves review quality — is untestable until phase 2 is accepted.

:comment[Fix: Add a phase 1 success criterion that verifies agent output quality is at least as good as the current skill on a reference document (e.g., run both on RFC-001 itself and compare finding counts). This does not require a full eval suite — a single calibration run is sufficient as a smoke test.]{#comment-1774990663287 text="approve"}

**\[Suggestion] hook-bundled-in-agent-rfc** (confidence: 82, quality)

The ExitPlanMode hook is a distinct deliverable bundled into an RFC nominally about reviewer agent architecture.

> "ExitPlanMode hook (PreToolUse) that suggests preflight doc formats when exiting plan mode in a scaffolded project"

The hook has a different trigger mechanism (PreToolUse vs. skill invocation), different implementation artifact (`hooks/hooks.json`, `hooks/exit-plan-mode.sh`), and different behavioral surface (fires automatically in all sessions vs. invoked by user). A reviewer who agrees with the agent architecture but has concerns about the hook — or vice versa — cannot accept one without accepting the other.

**Consequence:** The resulting ADR will authorize both features atomically. If the hook proves noisy and is reverted, the ADR record conflates its removal with the agent architecture. Minor, given the hook is advisory-only and trivially reversible.

:comment[Fix: Either split the hook into a separate RFC (RFC-002), or add a sentence in Meta explicitly noting the hook is bundled here because it was co-designed with the review skill orchestration and shares the same phase 1 delivery. The note makes the bundling intentional rather than accidental.]{#comment-1774990688130 text="yeah, that should have been a separate rfc - I scope-mashed."}

## Strengths

1. **Problem Statement is specific and measurable.** Three distinct problems are each grounded in concrete evidence: line counts, named external references with specific threshold values, and a named comparable project. This is the standard RFC-R06 should set for problem statements.
2. **Phased rollout with clean rollback.** Phase 1 being purely additive — agents exist but are not invoked — means zero behavioral regression risk and a trivial rollback path. The rollback plan is explicit and mechanical, not "we'll figure it out."
3. **Alternatives treated seriously.** All three alternatives include a "When to reconsider" clause. The shell script alternative correctly identifies it could supplement but not replace LLM review. This demonstrates the RFC author tested the alternatives against the actual constraints rather than dismissing them.

## Bogey Reviewer — Full Layer Output

### Layer 1 — Cross-doc assumption conflicts

**Conflict: RFC confidence tier vs. checklist-reviewer.md confidence tier**

- In RFC (Proposed Solution): Critical maps to confidence \= 100 only. Important maps to confidence 90–99.
- In `agents/reviewers/checklist-reviewer.md` (already committed): Critical maps to Error-severity rules at confidence ≥95. Important maps to Error-severity rules at 80–94, or Warning-severity rules at ≥90.
- Classification in RFC: SOFT (policy choice about tier boundaries)
- Classification in checklist-reviewer: SOFT (implementation of that policy)
- Why incompatible: The RFC is the authoritative spec for the system being built. The agent is a delivered artifact of that spec. They specify different thresholds for the same classification decision. Both are SOFT, but a SOFT constraint in a spec and a SOFT implementation of a different constraint cannot both be authoritative.

No other cross-doc conflicts identified. The RFC's FR references (FR-017 through FR-020, NFR-001, NFR-004, NFR-005) are all confirmed present in requirements.md. The RFC's phase 2 ADR-004 dependency is appropriately scoped as a future requirement. Constitution CONST-QA-01 tension with phase 1 success criteria is resolved by the additive-only nature of phase 1 (agents not invoked \= no changed skill behavior \= QA-01 not triggered until phase 2).

### Layer 2 — Hypothesis investigation

**H1 — Unverifiable requirement**

Not fired. The RFC delegates heuristic specification to agent prompt files, which are separate artifacts. The RFC's own success criteria are measurable (file existence, test passage, ≥85% accuracy in phase 2). The quality heuristic question is about agent file authoring quality, not RFC spec verifiability. Checked: no unverifiable requirements found in the RFC text itself.

**H2 — Silent conflict**

Fired. See confidence tier conflict above. The RFC's Proposed Solution section and the already-committed checklist-reviewer.md specify incompatible Critical/Important tier boundaries without either document acknowledging the discrepancy. Neither document cites the other as authoritative on this specific point.

**H3 — False hard constraint**

Not fired. "Seven agents, one per doc type" is correctly handled as a SOFT architectural choice with documented alternatives and explicit reconsideration criteria. The RFC does not foreclose single-agent approaches permanently — it scopes them out with stated conditions for revisiting. No mechanism is specified as mandatory where an outcome specification would suffice.
