# Universal Rules — All Document Types

| Rule ID | Rule | Severity | Applies to |
|---------|------|----------|------------|
| UNIV-01 | Document SHALL have YAML frontmatter with: status (from defined enum for that doc type), date (ISO 8601), owner, and version | Error | Durable governance docs only (see exemption below) |
| UNIV-02 | All cross-references SHALL use document/requirement IDs, not page numbers or prose | Warning | All |
| UNIV-03 | No section SHALL be entirely empty without a "TBD" marker and owner | Warning | All |
| UNIV-04 | Vague adjectives (fast, easy, simple, robust, scalable) SHALL be accompanied by quantification | Warning | All |
| UNIV-05 | Document SHALL NOT contain TODO/FIXME/HACK without an assigned owner | Warning | All |

## UNIV-01 doc-type scoping

UNIV-01 applies to **durable governance documents** that outlive any single feature branch:
`requirements`, `constitution`, `architecture`, `adr`, `rfc`, `test-strategy`, `interface-contract`.

UNIV-01 does **NOT** apply to **transient per-feature documents** generated by spec-kit:
`spec`, `plan`, `tasks`.

**Why:** Transient spec-kit documents are scoped to a single feature branch and PR. Their lifecycle
(status, ownership, versioning, date) is fully captured by the branch and PR metadata, and spec-kit's
templates use a prose metadata convention (`**Branch** | **Date** | **Spec**`) instead of YAML
frontmatter. Enforcing UNIV-01 on these docs produces uniform false positives without adding governance
value.

**How to apply:** When reviewing a document, first determine its doc type (per the orchestrator's
identification step). If the type is `spec`, `plan`, or `tasks`, skip UNIV-01 entirely — do not emit
a finding for missing frontmatter. For all other doc types, enforce UNIV-01 as written.
